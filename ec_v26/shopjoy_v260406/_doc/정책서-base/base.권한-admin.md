# 관리자(Back Office) 권한 정책

## 1. 권한 주체 구분

관리자 시스템에는 두 종류의 사용자가 존재한다.

| 구분 | 테이블 | 로그인 | 역할 접두어 |
|---|---|---|---|
| **내부 관리자** | `sy_user` | loginId + password | 접두어 없음 (ADMIN, OPERATOR 등) |
| **업체 사용자** | `sy_vendor_user` | sy_user에 연결 (`user_id`) | SITE_ / SALE_ / CS_ / DLIV_ / PROG_ |

두 주체 모두 `sy_role` → `sy_role_menu` RBAC 구조를 공유하되, 역할 접두어로 격리.

---

## 2. 공통 권한 모델 (RBAC)

```
sy_user / sy_vendor_user
  └─ sy_user_role (M:N)
       └─ sy_role
            └─ sy_role_menu (M:N)
                 └─ sy_menu
```

- 한 사용자에게 복수 역할 할당 가능
- 복수 역할 권한 충돌 시 **합집합(Union)** 원칙
- 권한 캐시: 5분 유지, 역할 변경 시 즉시 초기화

---

## 3. 내부 관리자 역할 (sy_user)

### 3.1 기본 역할 정의

| 역할 코드 | 역할명 | 권한 범위 |
|---|---|---|
| `SUPER_ADMIN` | 슈퍼관리자 | 전체 시스템 (사용자/역할 관리 포함) |
| `ADMIN` | 관리자 | 회원/상품/주문/배송/정산 전반 |
| `OPERATOR` | 운영자 | 주문/배송/고객센터 |
| `PRODUCT_MANAGER` | 상품담당자 | 상품/카테고리 |
| `SALES_MANAGER` | 판매담당자 | 주문/배송 |
| `MARKETING_MANAGER` | 마케팅담당자 | 쿠폰/이벤트/전시 |
| `SETTLE_MANAGER` | 정산담당자 | 정산/수수료 |
| `CUSTOMER_SERVICE` | 고객서비스 | 클레임/채팅/문의 |

### 3.2 메뉴 권한 플래그

각 역할마다 메뉴(`sy_menu`)별 4가지 권한:

| 권한 | 필드 | 설명 |
|---|---|---|
| 조회 | `can_read` | 목록/상세 조회 |
| 생성 | `can_create` | 신규 등록 |
| 수정 | `can_update` | 데이터 변경 |
| 삭제 | `can_delete` | 삭제/비활성화 |

### 3.3 역할 할당 규칙

| 규칙 | 내용 |
|---|---|
| 할당 주체 | 부서장 이상 또는 ADMIN/SUPER_ADMIN 보유자 |
| 상향 할당 금지 | 자신보다 상위 역할 할당 불가 |
| SUPER_ADMIN 할당 | SUPER_ADMIN만 가능 |
| 역할-메뉴 매핑 수정 | SUPER_ADMIN만 가능 |

## 4. 업체 사용자 역할 (sy_vendor_user)

업체 사용자는 `sy_vendor_user.user_id`로 `sy_user`에 연결되어 로그인한다.  
역할은 업체 유형에 맞는 **접두어 역할**만 할당 가능 — 타 접두어 역할 할당 불가.

### 4.1 업체 유형별 역할 접두어 요약

| 업체 유형 | vendor_class_cd | 역할 접두어 | 데이터 격리 기준 |
|---|---|---|---|
| 사이트운영업체 | SITE_* | `SITE_` | `site_id` |
| 판매업체 | SALE_* | `SALE_` | `vendor_id` |
| 고객상담업체(콜센터) | CS_* | `CS_` | 담당 채널 범위 |
| 배송업체 | DLIV_* | `DLIV_` | `courier_cd` |
| 프로그램관리업체(유지보수) | PROG_* | `PROG_` | 계약 범위 시스템 |

## 5. 사이트운영업체 권한 (SITE_)

## 10. 업체 권한 공통 정책

### 10.1 auth_yn 이중 제어

모든 업체 유형에 공통 적용:

| 구분 | 제어 대상 | 관리 주체 |
|---|---|---|
| `role_id` (역할) | 메뉴 접근, 기능 실행 | 시스템 관리자가 역할-메뉴 매핑 설정 |
| `auth_yn = Y` | `sy_vendor` 기본정보 수정 가능 여부 | 관리자가 개별 부여 |

- `auth_yn = Y`이더라도 역할(`role_id`)이 없으면 메뉴 접근 불가
- 총괄급 역할(`*_ADMIN`)이더라도 `auth_yn = N`이면 업체 기본정보 수정 불가

### 10.2 역할 교차 할당 금지

```
SITE_ ↔ SALE_   ❌ 교차 불가
SITE_ ↔ CS_     ❌ 교차 불가
SITE_ ↔ DLIV_   ❌ 교차 불가
SALE_ ↔ CS_     ❌ 교차 불가
...
내부 관리자 역할(ADMIN 등) ↔ 업체 역할(*_*)  ❌ 교차 불가
```

단, 한 `sy_user`가 여러 업체의 `sy_vendor_user`에 연결 가능 (업체별 별도 역할 지정).

### 10.3 권한 회수 트리거

| 트리거 | 처리 |
|---|---|
| 업체 `CLOSED` | 전 담당자 `user_id` 연결 일괄 해제 |
| 담당자 `LEFT` | `user_id` 연결 해제 (선택), 재활성화 불가 |
| `role_id = NULL` | 메뉴 접근 즉시 차단 |
| `vendor_user_status_cd = SUSPENDED` | 로그인 차단 |

### 10.4 권한 부여 절차 (공통)

```
1. 관리자(ADMIN 이상)가 sy_vendor_user 등록·수정
2. role_id 지정 (해당 업체 유형 접두어 역할만)
3. 필요 시 auth_yn = Y 부여 (총괄급만)
4. user_id 연결 → 로그인 활성화
```

PROG_ 업체 사용자는 SUPER_ADMIN만 등록 가능.

## 11. 권한 적용 흐름 (런타임)

```
사용자 로그인
  → accessToken에 user_id 포함
  → 서버: user_id → sy_user_role → sy_role → sy_role_menu (캐시 5분)
  → 클라이언트: 메뉴/버튼 표시 여부 결정

API 요청
  → Authorization: Bearer {accessToken}
  → 서버: token 검증 → 역할 확인 → 데이터 격리 검증 (vendor_id / site_id)
  → 권한 없음: 403 Forbidden → sy_api_log 기록
```

## 12. 관리자 화면 권한 표시 (UX)

### 12.1 상단바 역할 표시 및 전환

```
[상단바 우측]  👤 홍길동 · ADMIN  ▼
              ┌──────────────────────────────┐
              │ ✓ ADMIN (내부 관리자)         │
              │   OPERATOR                   │
              └──────────────────────────────┘

업체 사용자 예시:
[상단바 우측]  👤 김담당 · SALE_ADMIN  ▼
              ┌──────────────────────────────┐
              │ ✓ SALE_ADMIN [판매업체]       │
              │   SALE_ORDER                 │
              └──────────────────────────────┘
```

- 복수 역할 보유 시 드롭다운 역할 전환
- 전환 즉시 좌측 사이드바 메뉴 및 버튼 가시성 재계산

### 12.2 메뉴 필터링

현재 역할에 `can_read` 권한 있는 메뉴만 표시.  
권한 없는 항목은 숨김 처리 (dim 처리 아님).

### 12.3 버튼 가시성

```html
<button v-if="hasPermission('create')" class="btn btn-primary">신규 등록</button>
<button v-if="hasPermission('delete')" class="btn btn-danger btn-xs">삭제</button>
```
